# 社区动态写作指南

> 类别：`community/`（社区动态 / 运营月报 / SIG 公告，66 篇）
> 维护：随着文章迭代持续更新

---

## 一、受众分析

| 层级 | 角色 | 核心诉求 |
|------|------|---------|
| 主力读者 | 社区贡献者（现有/潜在） | "最近发生了什么，我能参与什么" |
| 次要读者 | 社区治理参与者（TC/PMC/SIG maintainer） | "决策透明度、社区健康度" |
| 外围读者 | 行业观察者 / 竞品社区 | "这个社区的活跃度、成长速度" |

这类读者有明确的行动潜质——他们需要的是**信息 + 参与入口**，而非被动消化的内容。

---

## 二、业界最佳实践

| 标杆 | 核心手法 |
|------|---------|
| **Kubernetes SIG Release Notes** | 每个 SIG 独立写 Release，结构化模板，一句话总结 + 可点击的详情链接 |
| **CNCF Project Updates** | TOC 月度简报，固定格式：新增项目 / 毕业项目 / 生态数据 / 活动预告 |
| **Rust Blog "This Month in Rust"** | 大量**外部链接**指向社区成员的贡献，而非自我宣传；每个链接有简短描述 |
| **Apache 软件基金会博客** | 明确区分"社区新闻"（客观）和"观点"（署名个人），不混为一谈 |
| **GitHub Octoverse 报告** | 数据可视化极强：一句话 headline + 一张图表，看到即理解 |

### 核心差异：self-promotion vs. curation

差的社区动态 = **"我们做了什么"的公告墙**——连续自我宣传，没有外部视角。
好的社区动态 = **"社区里发生了什么"的策展**——链接到社区成员的贡献、外部讨论、上游依赖的更新，像一份 newsletter 而非一份述职报告。

## 三、写作重点与笔法

### 月报

- **开头一句话总结**：「11 月，社区新增 3 个 SIG、软件包突破 X 个、贡献者首次超过 Y 人。以下是详细数据。」
- **数据可视化高于文字描述**：能用一张折线图/柱状图说清楚的指标别用一段文字。推荐 mermaid/echarts 渲染而非截图。
- **每个板块有"参与入口"**：不是「SIG X 成立」，而是「SIG X 成立，欢迎加入：<链接>，下一次例会时间：<时间>」。
- **排名/榜单有解释**：「本月 top 5 贡献者：张三（kernel 62 commits, 主要在 ARM64 errata）」——光有人名没意义。

### SIG 公告（成立/合并/招新）

- **三段模板**：
  1. 为什么需要这个 SIG —— 1-2 句说明背景（不要假设读者已经知道）
  2. 这个 SIG 做什么 —— 职责范围，边界明确
  3. 怎么参与 —— maintainer 介绍、例会时间、邮件列表、代码仓库链接
- **SIG 合并公告要解释动机**：不是「A 和 B 合并成 C」，而是「A 和 B 的职责重叠、各自人手不足，合并为 C 后能更好集中力量做 X」

## 四、当前 openEuler 文章的待改进点

### 问题一：月报太长，缺乏扫描入口
- **现象**：`社区 2022 年 11 月运作报告`——18,786 字符、25 张图，大量数据堆砌
- **改进**：在最顶部放一个「3 个数字」综述（一行读完）；每板块配一句关键结论加粗

### 问题二：SIG 公告像公文，缺乏人情味
- **现象**：「经 openEuler 技术委员会讨论决定，正式成立 X SIG」——机器人腔
- **改进**：加入 SIG 发起人的一句话（为什么想做这个 SIG、他/她的背景），让公告有面孔

### 问题三：缺少"参与入口"
- **现象**：很多 SIG 公告结尾没有例会时间、邮件列表、repo 链接
- **改进**：每个社区动态必须有一个明确的 "下一步行动" 链接

### 问题四：66 篇太少——大量 SIG 动态没被记录
- **现象**：2019-2026 年间仅 66 篇，社区里实际发生的变化远多于此
- **改进**：每个 SIG 至少 q 一篇年度总结 + 新 SIG 成立/重要变更必须发公告

## 五、改进方向

| 优先级 | 方向 | 具体措施 |
|--------|------|---------|
| 🔴 P0 | 月报加"一句话综述" | 每篇月报顶部一行读完的月度总结（3 个关键数字） |
| 🔴 P0 | 所有 SIG 公告加参与入口 | 例会时间 + 邮件列表 + 仓库链接，写到模板里强制执行 |
| 🟡 P1 | 月报数据图表化 | 用 mermaid/echarts 替代截图，让数据可引用可对比 |
| 🟡 P1 | 增加 SIG 年度总结 | 每个 SIG 每年至少一篇"今年做了什么，明年计划什么" |
| 🟢 P2 | 引入外部策展 | 月报不只写 openEuler 仓库的变化，还策展上游依赖的更新、相关开源项目的进展 |

## 六、自查清单

- [ ] 标题是否有具体时间/事件（不是泛泛的"社区动态"）？
- [ ] 是否有"一句话总结"让扫读的人 5 秒知道内容？
- [ ] 每个公告是否有明确的参与入口（链接/时间/联系方式）？
- [ ] 是否有人情味（至少一处真实的人名、引言或第一人称叙述）？
- [ ] 月报：数据是否有图表而非表格堆砌？排名是否有解释而非光有名字？
