# 生态/合作写作指南

> 类别：`ecosystem/`（生态 / 合作 / 商业案例，123 篇）
> 维护：随着文章迭代持续更新

---

## 一、受众分析

| 层级 | 角色 | 核心诉求 |
|------|------|---------|
| 主力读者 | 潜在生态伙伴（ISV/IHV/SI） | "他们也在用 openEuler，我能获得什么收益？怎么加入？" |
| 次要读者 | 企业客户 / 技术采购方 | "有没有和我同行业的客户已经验证过 openEuler？" |
| 外围读者 | 行业分析师 / 媒体 | "openEuler 的生态覆盖到什么程度了？" |

---

## 二、业界最佳实践

| 标杆 | 核心手法 |
|------|---------|
| **AWS Partner Network 案例** | 谁 + 解决了什么问题 + 用了什么服务 + 取得了什么结果——四段式，每段 2-3 句，配一张架构图 |
| **CNCF End User Case Studies** | 用户用自己的话讲；每个案例有"挑战→方案→效果→下一步"的独立叙事 |
| **Red Hat Customer Success** | 标题是客户的一句话（带引号的真实引用），而非官方的"XX 携手 YY" |
| **Google Cloud Case Study** | 每个案例有硬数据（降低了 X% / 加快了 Y 倍），不写形容词 |
| **Gartner Peer Insights** | 用户评分 + 结构化反馈——好的 case study 像一份微型审计报告 |

### 核心差异

差的生态文章 = **"双方签约合影 + 董事长致辞"**——这是 PR 通稿，不是给技术决策者看的内容。
好的生态文章 = **"一家什么公司，遇到了什么具体问题，为什么选了 openEuler，具体怎么做的，最后得到了什么可量化的结果"**——这是一个独立可传播的商业故事。

## 三、写作重点与笔法

### 伙伴加入/互认证公告

- 用「谁 + 为什么 + 怎么加入 + 之后计划做什么」替代「双方宣布合作」
- 伙伴名称首次出现时，一句话说清楚它是做什么的（不要假设读者认识）
- 互认证公告至少包含：测试环境（硬件型号/OS 版本）、测试结论、认证编号

### 商业案例/实践案例

**五段叙事结构**：
1. **客户是谁 + 面临什么挑战**（1-2 段，必须具体，不能泛化为"数字化转型中的挑战"）
2. **为什么选 openEuler**（真实原因：技术/成本/生态/安全，至少给一个具体论据）
3. **怎么做的**（架构概览 / 迁移方案 / 关键决策，1-2 段 + 一张图）
4. **效果**（必须有数据——"上线时间缩短 X%"比"显著提升效率"有用 100 倍）
5. **客户的一句话**（真实引用，不是 PR 浇灌过的）

### 行业报告/白皮书解读
- 先给一条最反直觉或最具信号意义的发现（抓住注意力）
- 数据和图表优先，文字只做解释
- 标题用数据而非形容词：「欧拉系产品整体装机量 102 万，金融行业占比第一」比「openEuler 行业领先」好 10 倍

## 四、当前 openEuler 文章的待改进点

### 问题一：篇幅太薄，全是通稿
- **现象**：「生态满足度+1 ｜ 湘江鲲鹏完成兼容性认证」——全文 709 字符，1 张图，只有一句"整体运行稳定，性能良好"
- **影响**：无法给正在评估 openEuler 的技术决策者提供任何有用信息
- **改进**：认证文章至少 1500 字符，包含测试环境和结论

### 问题二：缺少量化结果
- **现象**：「显著提升效率」「大幅降低成本」「实现了 AI 能力升级」——没有数字
- **改进**：必须附一个可公开的数据（哪怕是百分比区间），没有数据不发

### 问题三：缺少客户视角
- **现象**：每篇都是 openEuler 社区视角叙述，客户只是一个名字
- **改进**：引入客户自己的声音——一句话的带引号引用，或者客户方技术负责人的署名

### 问题四：标题全是"XX 加入 openEuler 社区"的模板
- **现象**：123 篇里大量同质标题
- **改进**：标题写伙伴的行业/技术特色而非"加入"动作本身——「金融 IT 龙头恒生电子发布基于 openEuler 的证券场景方案」比「恒生电子加入 openEuler 社区」更有信息量

## 五、改进方向

| 优先级 | 方向 | 具体措施 |
|--------|------|---------|
| 🔴 P0 | 案例必须带数据 | 所有商业案例/认证公告必须有至少一个可量化指标 |
| 🔴 P0 | 认证公告提高最低篇幅 | 不低于 1500 字符 + 包含测试环境 + 结论 |
| 🟡 P1 | 建立"客户署名"机制 | 有条件的案例文章让客户方技术负责人署名 |
| 🟡 P1 | 标题去模板化 | 从"XX 加入社区"改为"XX 在什么领域做什么" |

## 六、自查清单

- [ ] 标题是否有伙伴的行业/技术特色（不只是"加入"）？
- [ ] 是否讲清楚这家公司是做什么的（一句话）？
- [ ] 是否有至少一个可量化结果（百分比/倍数/时间）？
- [ ] 是否有客户自己的声音（引用或署名）？
- [ ] 认证公告是否包含测试环境和结论？
- [ ] 全文是否 ≥ 1500 字符（非图片字数）？
