# 新闻/快讯写作指南

> 类别：`news/`（新闻 / 快讯 / 奖项 / 里程碑，26 篇）
> 维护：随着文章迭代持续更新

---

## 一、受众分析

| 层级 | 角色 | 核心诉求 |
|------|------|---------|
| 主力读者 | 行业观察者 / 媒体 / 分析师 | "openEuler 最近有什么值得关注的事件" |
| 次要读者 | 社区成员 | "我们取得了什么认可，有什么自豪的事" |
| 外围读者 | 潜在用户 / 合作伙伴 | "这个社区/项目是否在向上走"（信号价值） |

---

## 二、业界最佳实践

| 标杆 | 核心手法 |
|------|---------|
| **Apache 新闻板块** | 极简：谁 + 做了什么 + 为什么是里程碑。一段写完，不放图片。 |
| **CNCF Announcements** | 每条新闻有明确的"为什么这对用户/社区有意义"的段落 |
| **Linux Foundation Press** | 区分三种：Press Release（正式）/ Blog Post（观点）/ Member Spotlight（人物）——格式不同，不能拿 PR 当 blog 发 |
| **GitHub Blog** | 短新闻：300-500 字符，一句话 headline，不放图；长分析：独立专题 |
| **The Register / Hacker News 讨论** | 好新闻引发讨论——因为它说了**一件具体的事**，而不是"XX 宣布 YY" |

## 三、写作重点与笔法

### 快讯类（CVE / 合规 / 加入某组织）

**三段式**：
1. **发生了什么**——一句话（「openEuler 正式成为 OpenHPC 成员」）
2. **这意味着什么**——对社区和用户的影响（1-2 段——这是读者真正要的内容）
3. **下一步**——后续计划或参与方式

### 获奖类

- 奖项名称 + 颁发机构 + 评选标准（一句话让读者知道这个奖的分量）
- **把聚光灯从"我们"转向"他们"**——获奖致辞感谢具体的人/贡献者/合作伙伴，而非空泛的"感谢社区"
- 不要放 5 张领奖合影——1 张照片足够，可以是集体照

### 里程碑类

- 「装机量突破 X」「贡献者超过 Y」「DDU 达到 Z」——先给数字，再给解释
- 里程碑文章是**为数不多的可以适度使用叙事笔法**的新闻类型——因为它天然有一个"从哪走到哪"的故事
- 但必须有一张趋势图——数字不是看绝对值，是看走势

### 语言风格
- 新闻类克制形容词（"重磅""重大""里程碑式的"在读者眼里全是反信号）
- 用动词和数字说话：不写"取得了令人瞩目的成就"，写"装机量从 50 万增长到 102 万，增长 104%"
- **每篇至少有一句"为什么这对你意味着什么"**

## 四、当前 openEuler 文章的待改进点

### 问题一：26 篇偏少——大量外部认可未被记录
- **现象**：多名 maintainer 入选 Linux Kernel 存储模块 Maintainer、论文入选 RTSS、加入 OpenHPC 等重大新闻才 26 篇
- **改进**：建立"社区新闻线索收集"机制，重要里程碑不遗漏

### 问题二：获奖文章太像内部贺电
- **现象**：「热烈祝贺 openEuler 荣获 XX 奖」——像内部通报而非对外传播
- **改进**：把视角转为"这个奖为什么能拿，对生态伙伴意味着什么"

### 问题三：缺少"why it matters"段落
- **现象**：「openEuler 正式成为 CNA」——只说了事，没说为什么重要
- **改进**：每条新闻必须解释这对读者的影响

---

## 五、改进方向

| 优先级 | 方向 | 具体措施 |
|--------|------|---------|
| 🔴 P0 | 每条新闻加"why it matters" | 1-2 段解释对读者/生态的实际影响 |
| 🟡 P1 | 建立新闻判断标准 | 什么值得发 news——外部第三方认可 / 里程碑数字 / 标准/合规变化；什么不值得——内部季度目标完成 |
| 🟢 P2 | 引入数据可视化 | 里程碑类新闻配趋势图，不只是单点数字 |

---

## 六、自查清单

- [ ] 是否在一段内说清楚了"发生了什么"？
- [ ] 是否有 1-2 段解释"这对你意味着什么"（why it matters）？
- [ ] 形容词是否克制（动词和数字替代形容词）？
- [ ] 是否有合理的署名/来源（不全是"社区"这一集体署名）？
