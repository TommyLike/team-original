# 版本发布写作指南

> 类别：`releases/`（版本发布 / 特性解读，35 篇）
> 维护：随着文章迭代持续更新

---

## 一、受众分析

| 层级 | 角色 | 核心诉求 |
|------|------|---------|
| 主力读者 | 现有用户 / 运维 / SRE | "要不要升级？升级会影响什么？怎么升？" |
| 次要读者 | 技术决策者 / 架构师 | "新版本在哪些能力上有代际提升？" |
| 外围读者 | 行业分析师 / 媒体 | "能引出什么有报道价值的新特性？" |

---

## 二、业界最佳实践

| 标杆 | 核心手法 |
|------|---------|
| **Ubuntu Release Notes** | 严格区分：What's new（用户能看见什么变化）/ Known issues（诚实）/ Upgrade instructions（可操作） |
| **Kubernetes Release Blog** | 每篇都有一个 theme（"Kubernetes 1.30: Leading the Charge with AI"），不列特性清单，讲特性故事 |
| **Go Release Notes** | 极简：一句 breaking change + 链接到 proposal；新手友好：每个新 API 有最小可用示例 |
| **Redis Release Notes** | 每个版本一句话总结 + 按模块分类：性能提升 / 新命令 / 行为变更 / 安全修复 |
| **macOS Release Page** | 用户视角优先——从不写"内核版本号从 23.0 升级到 24.1"，写"你的 Mac 现在可以做到 X" |

### 核心原则

**Release Notes ≠ Changelog。** Changelog 是给开发者看的全量变更；Release Notes 是给用户看的**高亮 + 升级指南**。好 release notes 回答三个问题：

1. 这个版本的主题（big idea）是什么？
2. 和上一个版本比，用户能多做什么 / 什么变了？
3. 我要升级，需要注意什么？

## 三、写作重点与笔法

### 版本发布公告

**五段结构**：
1. **一句话概括版本主题**：「openEuler 24.03 LTS 的核心是 AI 原生——从内核到用户态全面为 AI 工作负载优化」
2. **Top 5 新特性**——每个特性 2-3 句（它是什么、为什么需要它、怎么开始用）
3. **升级指南**——从哪个版本可以升、怎么升、有什么 breaking change
4. **性能提升数据**——如果有 benchmark，放对比数据（版本、硬件、测试条件，可复现）
5. **获取方式 + 反馈渠道**

### 特性解读（单篇）

- 不是「X 特性介绍」——是「X 特性解决了一个什么你遇到过的真实问题」
- 每个特性配一个**最小可行示例**（5 行代码以内）
- 复杂特性放架构图（参考 `../../image-style.md`），不要只放操作截图
- 结尾链接到完整文档 / SIG 主页 / 相关源码路径

### 语言风格
- 面向用户写（"你可以在 ARM64 上…"），而非面向社区写（"openEuler 社区新增了对 ARM64 的支持"）
- Breaking changes 必须诚实、醒目——不要藏在文末
- 避免 PR 腔："推出""重磅""赋能""打造"——直接用动词和数字说话

## 四、当前 openEuler 文章的待改进点

### 问题一：过度使用新闻稿腔
- **现象**：「openEuler 24.03 LTS 正式发布，麒麟信安同步推出服务器操作系统 V3.6.1，共启繁荣发展新篇章」——PR 语言
- **改进**：拆成两篇——openEuler 自己发技术 release notes，生态伙伴发适配公告

### 问题二：升级指南缺失
- **现象**：大量 releases 文章没有写"怎么从旧版本升级"、"有什么不兼容变更"
- **改进**：每个版本发布必须带升级指南 + breaking changes 清单

### 问题三：特性解读不够深
- **现象**：「pkgship 是什么？」——只介绍了概念，没有示例、没有典型使用场景
- **改进**：每个特性解读包含 1 个真实场景 + 1 段可复现的命令序列

### 问题四：版本号命名在标题中不够醒目
- **现象**：标题里版本号不统一（"openEuler 24.03 LTS"、"欧拉 24.03" 混用）
- **改进**：统一使用 `openEuler XX.XX` 格式；LTS 必须大写

## 五、改进方向

| 优先级 | 方向 | 具体措施 |
|--------|------|---------|
| 🔴 P0 | 建立版本发布模板 | 固定五段结构（主题概括 + 5 大特性 + 升级指南 + 性能数据 + 获取/反馈） |
| 🔴 P0 | 每版必带 upgrade guide | 明确写出支持从哪些版本升级、怎么升、breaking changes |
| 🟡 P1 | 特性解读独立成文 | 不从版本发布大稿里拆，而是每个值得讲透的特性单独一篇 `technical/` |
| 🟡 P1 | 命名规范统一 | 标题统一用 `openEuler XX.XX (LTS/创新版) 正式发布` |

- [ ] 标题是否含版本号且格式统一？
- [ ] 开头是否有一句话概括版本主题？
- [ ] 是否列出了 Top 5 新特性（每个有 2-3 句说明）？
- [ ] 是否写了升级指南 + breaking changes？
- [ ] 是否有性能/benchmark 数据（带测试条件）？
