# openEuler 公众号文章分类改进总览

> 基于 1863 篇文章的分类审视,按 7 个类别汇总问题与改进方向。
> 每个类别的详细规范见对应子目录的 `写作指南.md`。

---

## 一、跨类别共性缺陷

以下是 7 个类别**共同**存在的问题,优先级最高:

| # | 共性问题 | 涉及类别 | P0/P1 |
|---|---------|---------|-------|
| 1 | **机构腔过重**——"openEuler 社区推出/宣布/发布"视角,而非"你(读者)可以/需要注意"视角 | 全部 | P0 |
| 2 | **标题不承载观点**——描述主题而非表达结论,读者无法从标题判断要不要点进去 | 全部（people 除外） | P0 |
| 3 | **图片不承担解释功能**——活动海报占封面位、架构图用全量截图、slides 代劳概念图 | technical / releases / ecosystem | P0 |
| 4 | **文章不独立**——每篇开头是微信公众号外壳（`# [标题](微信链接)` + `[OpenAtom openEuler](javascript:void(0))` + 时间戳 + 微信 CDN 图片）,去掉这些后在搜索引擎/RSS/GitHub/内部知识库中无法独立存活,表现为:标题不是真正的 heading、无作者署名、无来源链接、图片依赖微信防盗链、正文开头缺乏背景交代 | 全部 | P1 |
| 5 | **数据缺失**——"显著提升""大幅降低"替代了具体的百分比、倍数、时间对比 | ecosystem / technical / news | P0 |
| 6 | **参与入口缺失**——文章结尾没有"下一步你可以做什么"的链接或指引 | community / events / technical | P1 |

## 二、按类别的问题·改进对照

### 1. technical（技术文章,1125 篇）

| 问题 | 现象 | 改进方向 | P |
|------|------|---------|---|
| 直播预告混入技术分类 | 576 字符 + 3 张海报 = "技术文章" | 直播预告归 `events/`;值得沉淀的技术内容单独成文 | P0 |
| 缺少代码块 | 抽样中几乎无 ` ``` ` 块 | 任何操作类内容必须有可复制命令/配置 | P0 |
| 缺少 benchmark 与复现条件 | 性能/调优文章无硬件型号、OS 版本、测试条件 | 见 rule.md 第 4 节——"经得起你凭什么这么说的追问" | P0 |
| 标题无观点 | 「openEuler 内核系列 ｜ Linux 内核发展史」 | 标题含结论:"从 v2.6 到 v6.x：Linux 内核三大架构变革回顾" | P1 |
| 配图装饰化 | 封面用活动海报,正文用 slides 截图 | 技术图按 `image-style.md` 的 Visual Guide 风格；取图优先找论文原图/官方架构图 | P2 |
| 个人署名缺失 | "原创*XX*" 被公众号元信息淹没 | 尊重个人署名,鼓励第一人称叙事 | P2 |

### 2. community（社区动态,66 篇）

| 问题 | 现象 | 改进方向 | P |
|------|------|---------|---|
| 月报太长无扫描入口 | 18K+ 字符、25 张图,无"一句话总结" | 月报顶部放 3 个关键数字综述 | P0 |
| SIG 公告缺参与入口 | 公告结尾无例会时间、邮件列表、repo 链接 | 模板强制:例会 + ML + repo + maintainer 联系方式 | P0 |
| 机构腔 | "经技术委员会讨论决定,正式成立 X SIG" | 加 SIG 发起人的一句话(为什么做这个 SIG) | P1 |
| 篇数太少 | 7 年仅 66 篇,大量 SIG 动态未被记录 | 每个 SIG 至少一篇年度总结;新 SIG 成立/重要变更必须发 | P1 |
| 缺少外部策展 | 只写 openEuler 仓库变化 | 月报策展上游依赖的更新、相关开源项目进展 | P2 |

### 3. events（活动/直播,481 篇）

| 问题 | 现象 | 改进方向 | P |
|------|------|---------|---|
| 重通知轻价值 | "直播预告 ｜ XXX" 只有时间+主题+讲师名 | 开头加一句"听了你能得到什么" | P0 |
| 同一活动发多篇 | 预告 + "正在直播" + "直播回顾" 各一篇,内容重复 | 一场活动最多 2 篇:预告 + 回顾；取消"正在直播"帖 | P0 |
| 回顾缺乏可引用内容 | 活动回顾 = agenda 列表 + 照片 | 每篇回顾提炼 3-5 条可直接引用的 technical takeaway | P1 |
| 嘉宾信息太薄 | 只列名字+公司 | 加一句话背景(ta 贡献了什么方向) | P2 |

### 4. releases（版本发布,35 篇）

| 问题 | 现象 | 改进方向 | P |
|------|------|---------|---|
| 新闻稿腔过重 | "共启繁荣发展新篇章"——PR 语言 | 拆成技术 release notes + 生态伙伴公告两篇 | P0 |
| 升级指南缺失 | 没有"怎么从旧版升级""有什么 breaking change" | 每版必须带 upgrade guide + breaking changes 清单 | P0 |
| 特性解读不够深 | "pkgship 是什么?"——只介绍概念,无示例 | 每个特性配 1 个真实场景 + 1 段可复现命令 | P1 |
| 版本号格式不统一 | "欧拉 24.03""openEuler 24.03 LTS" 混用 | 统一 `openEuler XX.XX (LTS/创新版)` | P1 |

### 5. ecosystem（生态/合作,123 篇）

| 问题 | 现象 | 改进方向 | P |
|------|------|---------|---|
| 篇幅太薄 | "认证公告" 全文 709 字符,只有一句"运行稳定,性能良好" | 不低于 1500 字符 + 包含测试环境 + 结论 | P0 |
| 缺少量化结果 | "显著提升""大幅降低"——没有数字 | 必须有至少一个可公开数据(百分比/倍数) | P0 |
| 标题模板化 | "XX 加入 openEuler 社区"——123 篇大量同质 | 标题写伙伴的行业/技术特色 | P1 |
| 缺少客户声音 | 每篇都是社区视角叙述 | 引入客户引用或客户方技术负责人署名 | P1 |

### 6. news（新闻/快讯,26 篇）

| 问题 | 现象 | 改进方向 | P |
|------|------|---------|---|
| 缺少 "why it matters" | "openEuler 正式成为 CNA"——只说了事,没说为什么重要 | 每条新闻必须解释对读者的实际影响 | P0 |
| 篇数偏少 | 大量外部认可未被记录 | 建立新闻线索收集机制 | P1 |
| 获奖文章像内部贺电 | "热烈祝贺"——内部通报而非对外传播 | 把视角转为"为什么能拿,对生态伙伴意味着什么" | P1 |

### 7. people（人物/专访,7 篇）

| 问题 | 现象 | 改进方向 | P |
|------|------|---------|---|
| 严重欠产 | 1863 篇文章仅 7 篇人物故事（其中 2 篇是节日贺卡） | 每月至少 1 篇贡献者故事;"五分钟采访"低成本模板 | P0 |
| 缺乏技术细节 | "参与比赛充满成就感"——只在谈感受 | 必须含至少 1 段技术描述(贡献了什么 repo/bug/功能) | P0 |
| 节日特辑太薄 | "1024·致敬程序员" 全文 193 字符 | 节日用真实故事驱动,不发群发贺卡 | P1 |

## 三、改进优先级总览（按影响面排序）

| 优先级 | 改进项 | 涉及类别 | 预期效果 |
|--------|-------|---------|---------|
| 🔴 P0 | 所有文章去掉机构腔,用读者视角写 | 全部 | 读者留存率提升(当前公众号视角劝退技术读者) |
| 🔴 P0 | 技术文章增加代码块 + benchmark + 复现条件 | technical | 技术文章从"可读"变为"可用" |
| 🔴 P0 | 活动通知加价值承诺;取消"正在直播"冗余帖 | events | 减少无效推送,增加参会转化 |
| 🔴 P0 | 生态文章加量化结果 + 客户声音 | ecosystem | 给技术决策者可引用的评估数据 |
| 🔴 P0 | 每版发布必须带 upgrade guide + breaking changes | releases | 升级焦虑是用户不开级的第一阻力 |
| 🔴 P0 | 建立月度贡献者 story 栏目 | people | 贡献者转化最直接的杠杆 |
| 🟡 P1 | 标题观点化（全类别） | 全部 | 点击率提升(标题是内容的第一也是最后一道过滤器) |
| 🟡 P1 | 社区动态增加参与入口 + 外部策展 | community | 从"公告板"升级为"社区策展" |
| 🟡 P1 | 技术文章系列化 | technical | 降低新读者的入门门槛 |
| 🟢 P2 | 配图升级(Vidual Guide 风格) | technical / releases | 差异化竞争力——技术图解是 distro 文章最稀缺的资源 |
| 🟢 P2 | 培养个人作者声音 | technical / people | 长期:社区从机构品牌转向个人品牌矩阵 |
