# 活动/直播写作指南

> 类别：`events/`（活动 / 直播 / 会议通知与回顾，481 篇）
> 维护：随着文章迭代持续更新

---

## 一、受众分析

| 层级 | 角色 | 核心诉求 |
|------|------|---------|
| 主力读者 | 潜在参会者 | "这个活动值得我去吗？时间、地点、我能得到什么" |
| 次要读者 | 已报名参会者 | "议程、嘉宾、怎么去、有直播链接吗" |
| 存档读者 | 社区成员 / 行业观察者 | "这个社区搞的活动质量如何？嘉宾层次怎样？" |

---

## 二、业界最佳实践

| 标杆 | 核心手法 |
|------|---------|
| **KubeCon / CNCF** | 议程公开在前，嘉宾信息完整；"为什么这个议题重要" 每个 session 有独立说明 |
| **Apple 发布会邀请** | 极简、悬念：「还有一件事」——不堆信息，留想象空间 |
| **AWS re:Invent** | 活动回顾不写流水账，而是按主题聚合 session 亮点 |
| **Google I/O** | 活动前：keynote 时间 + 直播链接；活动后：按技术方向整理 session 回放清单 |
| **DevRel 经典实践** | "sell the value, not the logistics"——先讲参加后能得到什么，再讲时间地点怎么去 |

## 三、写作重点与笔法

### 活动预告

**四要素（按重要性排序）**：
1. **我去了能得到什么**（1-2 句）——这是读者决定"去不去"的核心信息
2. **议程亮点**——不要列全部 session 标题，挑 3 个最有吸引力的
3. **嘉宾/讲师**——名字 + 一句话介绍（背景关联，不要扔头衔列表）
4. **时间、地点、报名链接**——放到文末，扫描的人知道去哪找

### 活动回顾

- ❌ 不要写流水账：「先是 A 讲了 X，然后 B 讲了 Y…」
- ✅ 要按主题聚合：「本次 Summit 的三个关键词：AI 原生、多样性算力、嵌入式」
- 每个主题下有至少一句具体引用或数据，而不是泛泛的"大家热烈讨论"
- 放活动照片而非全是 slides 截图——照片有面孔，slides 没有
- **回放链接必须放在最显眼位置**（开篇或独立一节），不要埋在文末

### 直播预告

- 直播预告和配套技术文章分开发——
  - 直播预告归 `events/`（写清楚主题、时间、讲师）
  - 直播涉及的技术内容写一篇独立的 `technical/` 文章
- 直播回顾应做成「录播链接 + 关键 takeaway 3 条 + Q&A 精华」

## 四、当前 openEuler 文章的待改进点

### 问题一：重通知轻价值
- **现象**：「直播通知 ｜ XXX」只有时间、主题、讲师名——没有告诉读者"听了能得到什么"
- **改进**：每个活动通知开头加一句价值承诺

### 问题二：直播预告与活动回顾混为一谈
- **现象**：同一场活动的"预告"和"正在直播"发了 2-3 篇，每篇内容几乎一样
- **改进**：预告 1 篇 + 回顾 1 篇，中间不发"正在直播"的占位帖

### 问题三：回顾缺乏可引用内容
- **现象**：大量活动回顾就是 agenda 列表 + 现场照片，没有一句可以被人引用的观点
- **改进**：每篇回顾提炼 3-5 条可直接引用的 technical takeaway

### 问题四：嘉宾信息太薄
- **现象**：只列名字和所属公司
- **改进**：加一句话背景（"XX 是 Y 项目的 maintainer，主要贡献在 Z 方向"）

## 五、改进方向

| 优先级 | 方向 | 具体措施 |
|--------|------|---------|
| 🔴 P0 | 活动通知加价值承诺 | 每个活动帖开头加一句"参加你能得到什么" |
| 🔴 P0 | 取消"正在直播"帖 | 同一场活动最多 2 篇：预告 + 回顾；直播链接通过预告帖更新 |
| 🟡 P1 | 回顾帖标准化 | 每篇回顾：3-5 个主题级 takeaway + 录播链接 + 照片 |
| 🟢 P2 | 建立活动日历 | 系列直播/meetup 用统一模板，带统一的视觉标识 |

## 六、自查清单

- [ ] 读者能在 5 秒内理解"我去了能得到什么"吗？
- [ ] 嘉宾信息是否超出一个列表（有背景关联）？
- [ ] 活动回顾是否有可引用的 takeaway（不是流水账）？
- [ ] 录播/资料链接是否在显眼位置？
- [ ] 同一活动没有"即将开始"之类的冗余帖？
