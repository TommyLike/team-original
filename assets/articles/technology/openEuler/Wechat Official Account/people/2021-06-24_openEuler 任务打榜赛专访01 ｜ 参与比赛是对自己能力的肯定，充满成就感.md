姚志聪（@yaozc701）

云计算工程师 

**Q：欢迎志聪，先向大家介绍一下你自己**

A：openEuler 社区开发者们好，我是姚志聪，一名从事云计算业务的开发者。

因公司与华为有合作关系并且通过华为鲲鹏展翅伙伴计划与鲲鹏凌云伙伴计划，才开始接触鲲鹏生态。

在工作上，项目有改造的需求，像对服务器、操作系统、数据库、中间件的进行替换，其中操作系统尝试选择 openEuler，大约是在 2020 年年底开始接触 openEuler，学习 openEuler 相关知识。

在学习过程中，我熟悉了 openEuler 系统，使用过 iSula 容器、数据库、中间件的源码编译与安装，基于 openEuler 搭建部署过 K8S，尝试过 openEuler 系统镜像换标裁剪等等。

上个月在社区看到有 openEuler 任务打榜赛，于是就报名了，希望能在比赛的过程学习到更多的相关技能。

**Q：openEuler 任务打榜赛目前已经进行一段时间了，向大家展示一下你目前的战绩**

A：我目前做了大概有 17 题。目前已经有 14 个题被合入社区，目前积分是 66 分。

**Q：你完成的这些赛题都是哪些方面？**

A：我做的大部分是软件包引入的题，也有一些软件包安全编译选项修改的题目。

**Q：题目难度方面，你觉得怎么样，有没有感觉特别难？**

A：就软件包引入的题目而言，对于 Python 和 Perl 类的软件包是比较简单的，答题者只要会使用 pyporter 和 perlporter 工具自动生成相应的 spec，就没有什么难度。

当然了，工具生成的也并非是完全正确的，有些生成的 spec 可能存在构建依赖或者安装依赖缺失，这样在本地编译构建或者 OBS 上面构建也是会失败的，这种就值得答题者去深究发现问题。

软件包编译选项修改的题目做的不算多不予评价。

比较有难度的是软件包测试用例补充，这个需要答题者去学习软件包的命令，并且这些软件命令不常用，编写测试用例也不太好编写。

**Q：针对目前 openEuler 任务打榜赛的赛题赛制，你有哪些建议？**

A：第一点：题目难度与分值需要正相关。题目尽量类型多点，避免出现像 Python 和 Perl 软件包一个工具解所有题的情况。

其次，软件包加固测试的任务建议出个测试用例 testcase 的文件结构以及如何编写的教程，这样可以指导更多小白参与到贡献中去。

最后，就是 maintainer 太忙了，提交的 PR 往往被忽略，希望能够提高 PR 合入效率。

**Q：这些题目类型中，你最喜欢的是哪个？**

A：个人比较喜欢软件包引入的题型。

**Q：从你的角度来看，当前的赛题类型是否可以满足你的参赛需求，你希望增加哪类赛题类型？**

A：可以增加提 issue 的任务类型，答题者在社区中发现问题或者需求，可以到相应的仓库去提 issue，maintainer 根据 issue 的价值给予合理的分值。

**Q：你之前有参与过开源社区的贡献吗？**

A：参加过，今年 1-2 月份在 openEuler OpenStack SIG 中跟 Maintainer、Commiter 学习如何签署 CLA、如何补包、补包流程以及 gitee 的工作流等。学习到了挺多知识，也尝试参与他们基于 OpenStack 21.03 的 Victoria 版本的适配工作，最后是提交了一个组件的软件包，也成功合入了。

**Q：你觉得参与开源社区贡献，对你来说，最大的收获是什么？**

A：可以在社区中认识许多大佬，并从中学习到很多技能。

因为我自己是从事云计算行业的工作，对云原生、容器、虚拟化有一定的基础，在 iSula、openstack、Kubesphere 等 sig 中的学习，自己的知识能够得到应用理解同时学习到新技能。

**Q：这次参与 openEuler 社区贡献的体验怎么样？**

A：感觉挺棒的，我还记得第一次参与社区的贡献的时候，看到自己的 PR 能够被合入，是对自己能力的一种肯定，还是比较有成就感的。

**Q：通过这次参与 openEuler 任务打榜赛，你收获了什么？**

A：收获就是学习到了许多知识技能，当然还有奖品。

大赛的负责人都很认真，请教他们打榜任务的流程和问题都能很快得到回应和解释。

在软件引入任务中，我碰见了挺多问题的，比如工具生成的 spec 不完善，有出现多的依赖也有出现少的依赖，这样都会出现构建失败的情况。也有本地环境的原因导致构建失败的情况。还有本地构建成功，在 OBS 上构建失败的情况。

这些问题都值得自己去深究学习，最后也是请教大赛的相关负责人明白了问题所在，明白了如何去处理这种问题。这样提高了自己探索发现问题和解决问题的能力，也明白了相关的知识和技能。
