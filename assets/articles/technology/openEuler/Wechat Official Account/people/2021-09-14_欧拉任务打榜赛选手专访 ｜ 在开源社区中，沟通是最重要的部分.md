2013 年开始对编程感兴趣，最初的尝试是搭建自己的网站。到了 2017 年，我有了一些编程经验，同时也有了第一次向开源社区提交 PR 的经历。我现在是 Apache committer 和 Apache APISIX 的 contributor，开源持续贡献者。

今年也作为学生参与了 Google Summer of Code 活动。在国内参与了开源软件供应链点亮计划，2020 年我是学生，今年是导师。

我是在今年 8 月份才看到欧拉任务打榜赛这个活动的。在这个活动中我学习到很多，接下来我将从贡献，收获，建议和感谢四个部分跟大家分享我在欧拉任务打榜赛中的经验。

首先是贡献部分，我在这次活动中做的任务主要是软件包导入，任务总数约 49 个，其中 Golang 软件库的任务有 41 个，其他软件包导入有 8 个，按软件包仓库计约 231 个。

同时我也开发了一个脚本工具 go2rpm-for-github 用来自动生成托管在 GitHub 平台上 Golang 软件库的 SPEC 打包文件，现在也已经上传到 Gitee 平台上。

go2rpm-for-github 仓库链接

https://gitee.com/bzp2010/go2rpm

第二部分是我在参与活动收获，主要有两点。首先我提升了自己的能力，我之前没有做过软件打包的工作，在这次活动过程中熟悉了 RPM 打包的方法和流程。在解决第一个 Golang 软件包时，耗费了至少 6 小时做了基本方法的探索，如上游社区的 SPEC 文件，使用 go2rpm 工具生成 SPEC 文件等，最后探索出一份具有通用性的打包文件方案。

在处理其他软件包引入的过程中，还做了软件移植和打补丁修复编译错误的工作。

同时，我认为在开源社区中沟通是最重要的部分，在做这些任务时候，也得到了社区中老师们的耐心反馈和指导，帮助我改进自己的方案。这些都是我在本次活动中的收获。

第三部分是我对本次大赛和 openEuler 社区的建议。

首先是建议增加对社区活动的宣传，这次的任务打榜赛在五月时就已开始，我大概在八月初才在群聊中看到有人发出的活动介绍链接。之后又因为有一些事情耽误，导致才在最后两周里疯狂做任务。

第二，建议在活动中对领题增加更多的条件，减少发生占题不做的情况和对做题速度快的增加领题额度等。同时，建议优化题目审核流程，常会出现负责审核的 Maintainer 和题目提交时间无法同步，造成题目审核不及时。

最后一点建议是对 openEuler 开源社区的，在做软件包导入的过程中，常发现依赖不齐全的问题，甚至有一些软件包存在但存在运行问题而无法运行，还缺乏一些用于快速 RPM 打包的 SPEC 的宏定义，这无法帮助社区构建灵活可扩展的软件包体系，建议社区提升对基础软件包质量的把控。

第四部分是感谢，首先感谢 openEuler 社区提供了这次活动的机会，之后是社区老师的耐心帮助和运营同学的大力支持。最后，我还要感谢现在这个科技时代，可以让我这种非专业的学生，有机会学习技术，参与技术活动。
