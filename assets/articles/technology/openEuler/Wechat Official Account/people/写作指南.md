# 人物/专访写作指南

> 类别：`people/`（人物 / 专访 / 贡献者故事，7 篇）
> 维护：随着文章迭代持续更新

---

## 一、受众分析

| 层级 | 角色 | 核心诉求 |
|------|------|---------|
| 主力读者 | 潜在贡献者 / 社区新人 | "我能不能也成为那样的人？这条路怎么走？" |
| 次要读者 | 社区现有成员 | "我了解这个人/这个角色背后的故事" |
| 外围读者 | 技术管理者 / HR | "这个社区的人才质量如何？" |

人物文章是所有类别里**转化率最高**的类型——它直接回答"我能不能成为其中一员"这个问题。

---

## 二、业界最佳实践

| 标杆 | 核心手法 |
|------|---------|
| **The New Stack "Makers" series** | 每篇聚焦一个开源贡献者；不是简历式罗列，而是跟着他/她的时间线找到关键转折点 |
| **GitHub ReadME Project** | 用主人公自己的声音（大段引用），不替他/她总结；配图是真人照片，不是头像 |
| **Google Open Source "Stories"** | 每个故事有一个戏剧性转折（"我以为我做不到，直到…"），让读者代入 |
| **Kubernetes Contributor Spotlight** | 固定三段：为什么开始贡献 → 最有成就感的一件事 → 对想加入的人的一句话建议 |
| **"五分钟采访" 形式** | 快问快答：5-6 个固定问题，每篇 800-1500 字，制作成本极低，产出稳定 |

### 核心差异

差的人物文章 = **在简历上贴照片**（哪毕业的、在哪工作、参与了几个项目）——这是 LinkedIn 简介，不是故事。
好的人物文章 = **一个具体的人，遇到了一个具体的困难，做了某个具体的选择，结果改变了一些什么**——读者看完记住的是那个"转折时刻"，而不是那串头衔。

## 三、写作重点与笔法

### 专访/贡献者故事

**叙事结构：**
1. **一个钩子**——主人公的一句话（大号粗体引语）或一个反常识的场景
2. **怎么入坑的**——不是简历时间线，而是"第一次提交 PR 的故事"
3. **一次挫折和一次突破**——读者最想看的，恰恰是困难而非光鲜
4. **现在在做什么**——技术内容足够具体（贡献在哪个 repo、解决了什么技术挑战）
5. **给想加入的人的一句话**

### 节日/特辑类

- 1024 程序员节、感恩节等——不要做群发祝福卡片（193 字的 "1024·致敬程序员" 是浪费发稿额度）
- 可以做：一个你今年遇到的代码 bug 故事 / 提交日志里最让你笑的一个 commit message / 开源社区里你被人帮过的一件事
- **节日特辑的最佳形式不是"社区祝你 XX"，而是"看看这群人这一年做了什么"**

### 笔法
- **保留主人公的语音特征**——如果他说"这个 patch 我改了 18 版差点放弃"，不要给编辑成"该补丁经过多轮迭代最终合入主线"
- **第一人称 > 第三人称**——专访尽量用 Q&A 形式或大段直接引用
- **不避讳挫折和负面**——真实的奋斗比完美的人设更有感染力

## 四、当前 openEuler 文章的待改进点

### 问题一：7 篇——严重不足
- **现象**：1863 篇文章里只有 7 篇人物故事（0.4%），其中还有 2 篇是节日贺卡
- **影响**：潜在的贡献者看不见"像我一样的人"在社区里做了什么、怎么做的
- **改进**：每月至少 1 篇贡献者故事；用低成本的"五分钟采访"模板稳定产出

### 问题二：现存人物文章缺乏技术细节
- **现象**：「参与比赛是对自己能力的肯定，充满成就感」——只在谈感受，没有技术内容
- **改进**：必须包含至少一段技术描述（贡献了什么、解决了什么技术问题）

### 问题三：节日特辑太薄
- **现象**：「1024·致敬程序员」全文 193 字符，1 张图
- **改进**：节日用真实故事驱动，不发生在节日当天发贺卡

## 五、改进方向

| 优先级 | 方向 | 具体措施 |
|--------|------|---------|
| 🔴 P0 | 建立"月度贡献者 story"栏目 | 每月至少 1 篇，用固定模板降低制作成本 |
| 🔴 P0 | 每篇含技术细节 | 贡献了什么 repo / 解决了什么 bug / 用什么语言和工具 |
| 🟡 P1 | 引入"5 分钟采访"快问快答格式 | 成本极低（800-1500 字），可以每周产出 |
| 🟢 P2 | 节日不群发贺卡 | 节日用"群像故事"替代祝福海报 |

## 六、自查清单

- [ ] 是否有一个具体的人（不是"社区成员"这种模糊群体）？
- [ ] 是否有主人公的直接引语（带引号的真实话语）？
- [ ] 是否描述了主人公的至少一个技术贡献（repo / bug / 功能）？
- [ ] 是否有一次挫折或困难（不是"一帆风顺"的故事）？
- [ ] 是否给出了"对想加入的人的一句话"？
