## 在阅读之前

本文件不是独立规范——它是在两份**上游标准**之上的 **openEuler 特化层**：

| 上游标准 | 覆盖内容 | 位置 |
|---------|---------|------|
| **技术写作手册** | 选题 / 结构 / 讲解顺序 / 严谨性 / 风格 / 用图 / 审校 / 评分表 | [`../../rule.md`](../../rule.md) |

## 一、受众分析

### 读者画像

| 层级 | 角色 | 占比估算 | 核心诉求 |
|------|------|---------|---------|
| 主力读者 | 开发者 / 系统管理员 / SRE | ~55% | "怎么用、怎么配、怎么排查"——需要可复现的实操内容 |
| 进阶读者 | 架构师 / 技术决策者 | ~25% | "为什么这样设计、有什么取舍"——需要架构级洞察 |
| 入口读者 | 学生 / 技术转岗 / 潜在贡献者 | ~15% | "我能不能看懂，值不值得投入时间"——需要清晰的切入点 |
| 外围读者 | 技术管理者 / 行业分析师 | ~5% | "技术竞争力在哪"——需要高层次的总结与亮点提炼 |

### 阅读场景
- **主力场景**：碎片化阅读（通勤、午休），通过标题和代码块快速扫描判断是否值得细读
- **存档场景**：被搜索引擎 / 内部知识库检索，作为排障参考
- **分享场景**：被转发的链接需要 3 秒内让人明白"这篇讲什么，对我有什么用"

这些标杆是 rule.md 里原则的**具体实现**：

|------|------|---------|

| **Dan Luu** (danluu.com) | 性能 / 分布式 | 每条结论有实测数据支撑；长文但结构极度清晰——对应 rule.md 第 3 节「严谨性」 |
| **Cindy Sridharan** | 基础设施 | 从第一性原理推导；不堆术语——对应 rule.md 第 2 节「自底向上」 |
| **Maarten Grootendorst** | ML / AI | 大量一致性极强的配图；一句旁注解释"为什么"——对应 rule.md 第 6 节「用图解释」和 `../../image-style.md` |
| **阿里/腾讯技术博客** | 互联网工程 | 线上问题→排查→根因→修复→预防，有完整的"破案"叙事——这是 distro/kernel 类文章最可复用的模式 |

### 适合 openEuler 的四种成熟模式

| 模式 | 结构 | 适用主题 |
|------|------|---------|
| **问题驱动型** | 遇到的异常现象 → 排查过程 → 根因定位 → 修复方案 → 预防措施 | 内核 bug、性能回归、兼容性问题 |
| **概念拆解型** | 这个技术解决什么问题 → 最简版本 → 现实版加了什么复杂度 → 一个例子 | 分布式调度、虚拟化、安全机制 |
| **动手实践型** | 目标 → 环境准备 → 分步操作 → 验证结果 → 常见坑 | 安装配置、迁移、CI/CD |
| **深度剖析型** | 整体架构 → 关键路径 → 源码/数据结构解读 → 设计取舍 → 性能特征 | 编译器、容器引擎、内核子系统 |

## 三、用上游标准审视当前 openEuler 技术文章

以下是抽取 1125 篇文章中的代表作，逐维对照 rule.md 的诊断。

### 维度对照表

| rule.md 维度（checklist） | 当前状态 | 典型表现 | 改进优先级 |
|---|---|---|---|
| **1. 选题**（写卡壳过的地方，不写复述文档） | ⚠️ 部分达标 | 内核系列、A-Tune 调优文章有真实困惑驱动；但大量"手把手带你玩转"系列是文档复述型 | 🟡 P1 |
| **2. 结构**（扫描式阅读：小标题 + 代码块作路标） | ❌ 普遍不达标 | 抽样中几乎没有代码块；大段文字无小标题分区 | 🔴 P0 |
| **3. 讲解**（自底向上建立直觉） | ⚠️ 部分达标 | 深度剖析型文章有自底向上意识；安装教程型直接平铺步骤无前置概念 | 🟡 P1 |
| **4. 严谨性**（数据 + 可复现 + 边界条件） | ❌ 普遍不达标 | 缺少 benchmark 数据、复现环境说明、边界条件声明 | 🔴 P0 |
| **5. 风格**（匹配主题与受众，不照抄模板） | ⚠️ 机构腔偏重 | "我们 openEuler 社区推出了…"而非"你可以这样做…" | 🟡 P1 |
| **6. 用图**（一图一概念 + 颜色语义 + 旁注） | ❌ 普遍不达标 | 活动海报占封面位；架构图用全量截图而非拆解后的概念图 | 🟢 P2 |

### 具体问题的深度诊断

#### 问题一：结构不满足扫描式阅读（rule.md 第 2 节）

**现状**：抽样文章几乎**没有代码块**（` ``` `），大量长段落、图多字少。一篇技术文章 576 字符 + 3 张图——对搜索引擎和人类读者同等不友好。

**rule.md 的对照标准**（引自上游）：
> "小标题 + 代码块作为'路标'：让扫描的人能快速定位。"

**改进**：任何涉及操作的内容必须包含可复制的命令/配置片段；关键概念用粗体关键词而非段落描述。

#### 问题二：严谨性缺失（rule.md 第 4 节）

**现状**：文章缺少"你凭什么这么说"的支撑——没有 benchmark 数据、没有复现环境说明、没有边界条件声明。

**rule.md 的对照标准**（引自上游）：
> "一篇文章应该经得起'你凭什么这么说'的追问：有没有实测数据支撑？关键判断能不能复现？有没有说明边界条件？"

**改进**：内核调优/性能类文章必须带 benchmark（硬件型号 + OS 版本 + 测试条件），承认不适用的场景。

#### 问题三：选题偏"文档复述"（rule.md 第 1 节）

**现状**：大量"手把手带你玩转 openEuler"系列是官方文档的实操版复述，而非从一个**具体卡壳点**出发的独立文章。

**rule.md 的对照标准**（引自上游）：
> "好文章不是'我懂什么，来展示一下'，而是'我在某个具体问题上卡过壳，后来想通了，把它写下来帮别人'。"

**改进**：安装教程类文章应从"真实安装过程中最容易绊倒的 3 个坑"切入，而非遍历安装文档的每一个选项。

#### 问题四：配图不承担解释功能（rule.md 第 6 节）

**现状**：封面用活动海报、正文里堆 slides 截图。图是"放上去好看"而非"解释概念"。

**rule.md 的对照标准**（引自上游）：
> "一图一概念。不要在一张图里塞完整架构。颜色即语义——每个概念分配一个固定颜色。"

**改进**：需要建一张新架构图时，用 `../../image-style.md` 的 STYLE 块生成。不要用 slides 截图代劳。取图优先找真图（论文原图 / 官方架构图），没有真图才 AI 生成。

#### 问题五：标题不承载观点（rule.md 第 2 节）

**现状**：「openEuler 内核系列 ｜ Linux 内核发展史」——只描述主题，不给结论。

**rule.md 的对照标准**（引自上游）：
> "标题需要做到：有逻辑 + 有观点。一句话总结本页内容——作为标题核心原则。"

**改进**：标题应回答"这篇文章的结论是什么"——「从 v2.6 到 v6.x：Linux 内核三次最重要的架构变革」。

## 四、改进方向

| 优先级 | 方向 | 具体措施 | 对应上游 |
|--------|------|---------|---------|
| 🔴 P0 | 建立最低内容标准 | 每篇技术文章必须包含：1 个具体问题 / 1 段可复现的代码或配置 / 1 张解释概念的图（不包含封面） | rule.md 第 2、4、6 节 |
| 🔴 P0 | 增加代码块和复现条件 | 涉及操作的内容必须有可复制命令；性能/调优内容必须有 benchmark 条件 | rule.md 第 4 节 |
| 🔴 P0 | 直播预告与文章分离 | 直播预告归 `events/`；直播涉及的技术内容写成一篇独立的 `technical/` 文章（可事后发） | rule.md 第 1 节 |
| 🟡 P1 | 系列化写作 | 把分散的内核/虚拟化/容器/编译器文章串成系列，给读者清晰的进阶路径 | rule.md 第 3 节 |
| 🟡 P1 | 培养个人作者声音 | 鼓励个人署名和第一人称叙事（"我发现…""我试了…"），避免"社区"集体署名的机构腔 | rule.md 第 5 节 |
| 🟢 P2 | 配图升级 | 技术概念图优先用 `../../image-style.md` 的 Visual Guide 风格 | rule.md 第 6 节 |
| 🟢 P2 | 标题观点化 | 每个标题含一个结论或问题，不只描述主题 | rule.md 第 2 节 |

## 五、自查清单

写完一篇技术文章后逐条核对。**每一项后面括号里的引用是指向上游标准的来源。**

- [ ] 标题是否包含了一个明确的结论或问题？（rule.md 第 2 节）
- [ ] 开头是否回答了"这篇解决什么、读完能带走什么"？
- [ ] 内容是否自底向上排序——先简单版本，再加复杂度？（rule.md 第 3 节）
- [ ] 是否有至少一段可复制的代码/命令/配置？（rule.md 第 4 节）
- [ ] 关键判断是否有数据或来源支撑？（rule.md 第 4 节）
- [ ] 是否说明了边界条件——"这在什么前提下成立，什么情况下不成立"？（rule.md 第 4 节）
- [ ] 每张配图是否解释了"为什么这里重要"？（rule.md 第 6 节；配图生成 → `../../image-style.md`）
- [ ] 结尾是否给读者一把"能带走的钥匙"——下一步动作 / 记住一句话 / 下一篇链接？
- [ ] 文章能否独立于公众号被阅读？（去掉 header/footer 后是否仍然完整？）
- [ ] 是否有个人署名（而非只有"社区"集体署名）？
