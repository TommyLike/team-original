## 1. 选题：写"卡壳过的地方"

### 核心原则
好文章不是"我懂什么，来展示一下"，而是"我在某个具体问题上卡过壳，后来想通了，把它写下来帮别人"。前者天然带读者视角（你自己就是曾经的读者），后者容易变成知识炫耀或文档复述。

选题时反着问自己：
- 我在哪个**具体**问题上卡过？（不是"我觉得 X 很难"，而是"X 里的 Y 机制我一开始没理解"）
- 这个困惑有没有被现有内容清楚地解释过？如果没有，就是内容空白，值得写。
- 优先写"被低估、被冷落"的主题，而不是追热点。

### 正例 vs 反例
| | |
|---|---|
| ✅ 好 | 「为什么 1:1:1 的 Pod:Server:Kernel 是 code interpreter 的合理设计」——从一个真实困惑出发，有自己的推理 |
| ❌ 差 | 「XX 框架入门教程」——复述官方文档，没有增量信息 |

### 判断他人账号时
看它是在**填补具体困惑**，还是在**搬运/复述**官方文档和新闻。持续产出"解决真实困惑"的号，质量远高于信息聚合型的号。

## 2. 结构：为扫描式阅读设计

### 核心原则
大多数人不逐字读技术博客，而是**扫描标题和代码片段**去找自己需要的信息。所以结构要为"扫描"而非"线性精读"设计。

具体做法：
- **开头点题**：前一两段就交代清楚"这篇解决什么问题、读完你能拿到什么"。技术文章不适合把结论留到最后揭晓——没有好的钩子，读者不会留下来看结果。
- **给可预期的结构**：读者需要知道接下来会讲什么，中途突然转向会让人失去方向感（技术文章不是悬疑小说）。
- **小标题 + 代码块作为"路标"**：让扫描的人能快速定位。
- **复杂主题拆成系列**：与其塞进一篇巨长的文章，不如拆成 2-3 篇连载，每篇能一次读完。抵制"把所有东西堆进一篇"的冲动，用链接串联相关内容。

### 用人对人说话的口吻
写成"一个人在跟另一个人解释"，而不是写成一本操作手册。手册腔会劝退读者。

## 3. 严谨性：数据、可复现、承认不确定

> **本节是技术文章区别于"信息搬运"的核心，也是评估质量最有效的过滤器。**

### 核心原则
高质量技术长文的共同特征：把**系统性思考**和**严谨的测量**结合起来——带数据、带历史背景、带可复现的推理过程。

一篇文章应该经得起"你凭什么这么说"的追问：
- 有没有**实测数据**支撑结论，而不是"个人感觉"？
- 关键判断能不能**复现**？（复现步骤、代码、环境说明）
- 有没有说明**边界条件**？（在什么前提下成立，什么情况下不成立）

### 承认不确定性
明确区分"我确定的"和"我在推测的"。过度确定的结论是危险信号；坦诚说明局限反而增加可信度。一个好的技术写作者会清楚地划分自己有把握的领域和在猜测的领域。

### 关键过滤器
判断一个公众号/作者质量时，问一句：
> 他能不能提出一个**可验证的技术判断**，还是只在做泛泛而谈的信息搬运？

这一条基本可以替代"新榜/清博指数"式的流量榜单——流量高 ≠ 内容严谨。

## 4. 风格：匹配主题与受众，而非照抄

### 核心原则
**没有唯一正确的写作风格。** 很多"写作建议"本质是"你应该像我这样写"——写得短的人建议大家写短，这只是描述了某个人自己的风格，不是普适规律。

重要的是找到**适合你自己和你所处场景**的风格，而不是照抄别人。照抄别人的表面风格通常学不到位：容易复制表面（排版、语气、堆料），而错过驱动那个风格的底层逻辑。

> 类比：跳舞时只模仿老师的脚步动作、却不理解重心如何移动的人，反而会做错。技术写作同理。

### 评判时的含义
不要用单一的"文笔标准"去评判所有文章，而要看**风格是否匹配主题和受众**：
- 写 Triton 编译器内部机制的深度文章，理应比"三分钟入门"更长、更密——这不是缺点。
- 追热点、堆术语但没有实质内容，才是问题。

## 5. 用图解释：Visual Guide 手法

> 本节提炼自 Maarten Grootendorst 的 "A Visual Guide to ..." 系列（quantization /
> MoE / Mamba / reasoning LLMs / LLM agents / diffusion，原文已归档在
> `sources/`）。这类文章的核心竞争力不是文字，而是**用一连串一致、极简的图，
> 把抽象机制变得"看得见"**。

### 核心原则
抽象概念（注意力、路由、量化、扩散）最大的理解门槛是"脑子里建不起图像"。
好的技术长文用**大量小图**替读者建好这个图像——每张图只承担**一个**理解负担：
一次映射、一条数据流、一处对比。

具体手法：
- **一图一概念**：不要在一张图里塞完整架构。把复杂机制拆成 50–70 张小图，
  每张只推进一步（这几篇平均每篇 55–78 张图）。
- **颜色即语义**：给每个概念分配一个固定颜色，全文所有图 + 正文内联关键词都用
  同一个色。读者记住一次，全篇复用。
- **渐进式揭示**：同一个结构反复出现，每张图只点亮/改变一个部分，像动画的逐帧。
- **正文与图咬合**：正文关键词的颜色和图里对应元素一致，形成"文字↔图"的锚点。
- **箭头 + 旁注**：用一句话的旁注（配一根弧线箭头）解释"这里为什么"，而不是把
  解释堆在正文里。

### 正例 vs 反例
| | |
|---|---|
| ✅ 好 | 量化文章用「钟形分布 + 紫点落入 INT4 格子」一张图，让"多个权重映射到同一桶"一眼看懂 |
| ❌ 差 | 用一张塞满框和线的"完整架构图"，标注十几个模块，读者无从下眼 |

### 配图风格规范（重要）
这套视觉语言已单独整理成**可执行规范** + **可复用的生图提示词**，并已用 Gemini
`gemini-3-pro-image` 实测验证跨主题一致性：

> 📎 **见同目录 [`image-style.md`](./image-style.md)**
> —— 调色板（含 hex）、形状/排版语言、可直接拼接的 STYLE 提示词块、
> 生成接口与一致性自查清单。

在 **explore 流程**书写技术类文章、需要配图时，插图应按 `image-style.md` 的
STYLE 块生成，确保全文风格统一。

## 6. 审校：写完不等于发布

### 核心原则
写完只是第一步。稳定产出高质量内容的作者，背后往往有固定的审校/复盘习惯。

推荐流程：
1. **冷却再读**：写完隔一两天再读一遍，让大脑从写作状态里跳出来。
2. **敢于删**：不要爱上自己的文字。写得再好但不匹配主题的段落，该删就删。编辑花的时间，可能和写作一样多。
3. **双重审校**：
   - 找 **1-2 个该领域专家** 审技术正确性，避免事实性错误。
   - 找 **1 个目标读者** 审可理解性，确保新手也能看懂。
   - （若有需求方/业务方，也找一个人看，验证是否达成目的。）

### 作为质量代理指标
持续输出高质量的号，文章里很少出现前后矛盾、未经推敲的结论——这本身反映了背后有审校/复盘机制。评估账号时可以留意这一点。

## 7. 质量评分表（可打分）

> 用于评估候选账号，或审核团队自己产出的技术文章。可按 1-5 分打分。

| 维度 | 1 分（差） | 3 分（合格） | 5 分（优秀） |
|---|---|---|---|
| **选题** | 复述官方文档 / 新闻 | 有一定信息增量 | 从具体困惑出发，有独到解法 |
| **结构** | 意识流、无小标题 | 有基本分段 | 开头点题，代码/图表可扫描定位 |
| **严谨性** | 断言无依据 | 有部分佐证 | 有数据 / 复现步骤 / 边界条件说明 |
| **诚实度** | 结论过度确定 | 偶尔提及局限 | 清晰区分确定与推测，坦诚局限 |
| **风格匹配度** | 追热点堆术语 | 风格基本得体 | 深度和篇幅精准匹配主题复杂度 |

**评估建议**：
- 单篇文章：五维加总，关注是否有明显短板（某一维 ≤2 分需警惕）。
- 账号长期追踪：连续观察 1-2 个月的更新，看**均值**和**稳定性**，而非单篇高光。
- 配合 RSS 工具（如 WeWe RSS / we-mp-rss）批量订阅候选号后长期观察，比一次性榜单排名更能反映真实水平。

## 8. 发布后：分发与迭代

### 核心原则
发布 ≠ 结束。想让内容触达读者，要主动在目标读者聚集的地方分发。

- 选对发布平台：让内容有最大概率触达目标受众。
- 主动分发：在读者活跃的社区/群/平台分享。
- 收集反馈迭代：把读者的困惑和纠错，反哺到下一篇选题。
