# 开源项目研究维度提纲

> 用途:系统化地把一个开源项目**拆全、拆透**的研究清单(due diligence)。
> 目标是覆盖度(MECE),哪怕枯燥但必要的项(许可证、bus factor、威胁模型、依赖供应链)也不遗漏。

## 使用说明:先分清两个维度

本提纲只覆盖 **研究维度**,不涉及 **叙事维度**。二者目标不同,不要混用:

| | 研究维度(本提纲) | 叙事维度(另行处理) |
|---|---|---|
| 目的 | 把项目拆全、拆透 | 让读者读得下去、被说服 |
| 追求 | 覆盖度、MECE、客观 | 代入感、说服力 |
| 取舍 | 不遗漏 | 可裁剪、可重排 |

研究先做全,写作时再由叙事维度对内容做取舍与排序。

## 取材方法:有源码仓库的项目,必须本地克隆

纯 web 抓取(博客 / 论文 / README / API 元数据)对以下维度天然盲区:

- **② 原理**:架构细节、核心抽象、数据流——README 和博客只讲意图,真正的实现在代码里。
- **③ 质量**:测试覆盖率、代码规范、fuzz testing、依赖审计——不读代码只能标"未知"。
- **④ 可持续**:bus factor 是否比 git log 显示的更糟(比如所有核心贡献者其实都是同一公司员工)。

**凡是有公开源码仓库的项目,流水线必须在研究启动前做:**

1. **`git clone` 到 `reference/src/`**(只做 shallow clone `--depth 1`,不拖全历史)。
2. **读关键设计文档**(按优先级,**不必全读代码**,但以下文件必须翻:
   - `README.md`、`CONTRIBUTING.md`、`CHANGELOG.md`、`SECURITY.md`、`GOVERNANCE.md`、`CODEOWNERS`
   - `docs/` 下的架构/设计文档(至少前 5 篇)
   - `ARCHITECTURE.md` / `DESIGN.md` / `ROADMAP.md`(若有)
3. **跑一次 `npm audit / cargo audit / pip-audit / trivy fs`**(有的话,记录结果而非空口说"依赖审计未做")。
4. **用 `cloc` 或 `tokei` 记下代码规模**(总行数、测试行数、测试/源码比——一个数字胜过三段描述)。
5. **盘点测试基础设施**:有无 CI 配置文件、测试框架名、覆盖率 badge。
6. **找部署与运维线索**(新增,对应 ③c/③d):
   - 搜索 `Dockerfile`、`docker-compose.yml`、`Makefile`、`Justfile`、`Taskfile`——判断安装方式和部署拓扑。
   - 搜索 `helm/`、`k8s/`、`deploy/`、`terraform/` 目录——有无官方部署模板。
   - 搜索 `metrics`、`health`、`/healthz`、`prometheus`——有无可观测性埋点。
   - 搜索 `bench`、`benchmark`、`perf` 目录或脚本——性能数据是否可复现。
   - 搜索 `migration`、`upgrade`、`CHANGELOG` 中的 breaking change 标注——升级策略。

> 这些项目不要求研究员逐行审计源码——**目标是让 ② 原理 和 ③ 质量 从
> 「web 推测的轮廓」升级到「读代码后备书的具体判断」**。
> 如果某些检查跑不了(如私有仓库 / 无审计工具),在报告里诚实标注「无法验证」即可,而不是跳过整个取材步骤。

## 维度总览

研究维度分为两大半:

- **理解项目本身**:① 定位 · ② 原理
- **评估是否值得押注**:③ 质量 · ④ 可持续

---

## 一、理解项目本身

### ① 定位 · 是什么 / 为谁

核心问题:它精确地解决什么问题?为谁?边界在哪?

- [ ] 问题域与定位:它精确地解决什么问题?**不解决**什么?
- [ ] 目标用户与场景:谁、在什么场景下用它、替代了原来的什么做法?
- [ ] 能力边界:声称能力 vs 实际覆盖范围;官方明确的非目标(non-goals)。

### ② 原理 · 怎么实现

核心问题:它凭什么能做到?核心机制、依赖与扩展点。

- [ ] 架构与核心机制:核心抽象是什么、数据流 / 控制流怎么走、"魔法"发生在哪。
- [ ] 技术栈与依赖:语言 / 运行时;关键第三方依赖;上游依赖谁、下游被谁依赖(供应链)。
- [ ] 接口与扩展点:API / CLI / library 形态;配置模型;二次开发 / 插件的挂载点。

---

## 二、评估是否值得押注

### ③ 质量 · 好不好 / 可不可信

核心问题:工程可信度如何?能不能上生产?

#### ③a 工程可信度

- [ ] 测试与 CI:测试框架、覆盖率(有数字不要形容词)、CI 配置文件、fuzz testing / property-based testing 有无。
- [ ] 代码规范:语言 standard lint 规则、formatter 配置、类型覆盖(如 TypeScript strict mode)。
- [ ] 文档完备度:README 是否覆盖「是什么 / 怎么装 / 怎么用 / 怎么配」四问;有无架构文档;API 文档是自动生成还是手写。

#### ③b 安全态势

- [ ] 安全模型:项目自己声称的安全边界是什么?在哪画线?
- [ ] 威胁模型与审计:是否有正式威胁模型文档?是否有公开第三方安全审计?
- [ ] 漏洞记录与响应:CVE 历史、修补速度、披露透明度(有没有"静默修复")。
- [ ] 默认安全(secure-by-default):空配置/最小配置是否已经安全?还是依赖用户把每个选项配对?

#### ③c 性能与容量

- [ ] 基准数据:官方是否提供 benchmark?复现条件是否公开?有没有独立第三方复现?
- [ ] 资源占用:冷启动延迟、稳态内存、CPU 开销、磁盘 I/O——每个平台各一组数字。
- [ ] **容量上限与硬限制**:最大连接数 / 吞吐量 / 数据规模的硬顶在哪?超出后是优雅降级还是直接挂?
- [ ] **硬件最低要求与环境**:CPU 架构(amd64/arm64)、OS 版本下限、内存/磁盘最小需求、是否需要特殊内核特性(如 eBPF、user namespaces)。

#### ③d 生产就绪度

核心问题:假设技术选型的人已经决定用它,**怎么把它真正跑起来、跑稳**?

- [ ] **安装方式**:几种装法(npm/pip/apt/brew/docker/helm)?有没有 one-liner?哪个是推荐方式?
- [ ] **部署拓扑与运行依赖**:单进程还是需要外部服务(数据库/消息队列/etcd)?需要 root 吗?需要内核参数调优吗?有 sidecar/agent 吗?
- [ ] **配置模型**:几项必配、几项可选?默认值是否合理?有没有配置校验(如 Zod schema)?
- [ ] **平台兼容性矩阵**:每个 OS/arch 组合是「正式支持」「实验性」「明确不支持」中的哪种?不要把「声称跨平台」当成「三端同等成熟」。
- [ ] **升级与回滚**:升级怎么操作?有没有 migration?能不能回滚?breaking change 怎么通知?
- [ ] **健康检查与可观测性**:有没有 health endpoint?暴露 Prometheus metrics 吗?日志是结构化 JSON 还是 printf?支持 tracing?
- [ ] **已知限制与坑**:官方是否维护 known limitations 清单?社区踩过哪些坑(GitHub Issues 里的常见问题)?
- [ ] **生产参考架构**:有没有官方/社区提供的「X 规模推荐 Y 配置」参考?有没有人公开写过生产部署复盘?
- [ ] **API 稳定性承诺**:有 semver 吗?deprecation policy 是什么?实验性 API 怎么标注?

### ④ 可持续 · 活不活 / 值不值

核心问题:项目活得好不好?长期能否依赖?

- [ ] 社区与治理:贡献者规模与集中度(**bus factor**)、活跃度、决策机制、背后组织。
- [ ] 生态位与竞品:同类项目对比、差异化、可替代性、迁移难度。
- [ ] 许可证与商业模式:开源协议、专利 / 合规风险、商业化路径(决定长期是否可依赖)。
- [ ] 采用与运维:上手成本(从零到跑起来要几分钟)、迁移成本(从竞品迁过来要改多少)、锁定风险(有没有只能用它的专有接口)、支持渠道(社区/商业/无)。

## 权重提示(按项目类型调整)

不同项目簇的研究重点不同:

- **工具类(CLI / dev tool / 本地沙箱)**:重 ③d 生产就绪度(安装是否 one-liner、平台兼容矩阵、升级路径)和 ③c 性能容量(启动延迟、资源占用)。
- **平台 / 框架类**:重 ② 原理(扩展点、数据流)与 ④ 可持续(生态、治理)。③d 关注部署拓扑(是否需要外部依赖)。
- **基础设施类(数据库 / 消息队列 / 网络组件)**:重 ③b 安全态势、③c 性能与容量(硬限制、扩展天花板)、③d 生产就绪度(高可用部署、健康检查、备份恢复)、④ 治理(长期可依赖性)。
